## A. データ保全

### A-1. スタンプ取得データを機能改修で消さない
- **現状**: 来場者データは localStorage の `participant_id` + `recovery_code` で識別。DBのデータは消えないが、マイグレーションのカスケード削除（tier削除など）で `participant_rewards` が消える既知リスクあり（HANDOFF §3-3）。
- **要望**: 改修のたびにユーザーのスタンプ・特典データが失われないようにしたい。
- **検討事項**:
  - マイグレーション方針の明文化（削除系DDLは慎重に）
  - `participants` / `event_stamps` / `participant_rewards` に論理削除フラグを検討
  - 改修前後のデータ確認手順をドキュメント化

### B-3. デザインの微調整（細かいUI修正）

## C. ユーザー機能

### C-1. ユーザー名（ニックネーム）の変更
- **現状**: `/profile` ページはニックネームを**表示するだけ**で編集UIは未実装（`app/profile/page.tsx`）。
- **要望**: プロフィールページでニックネームをタップ・編集できるようにしたい。
- **実装内容**:
  - ニックネーム欄をタップでテキスト入力に切り替え（インライン編集）
  - 保存時は `lib/storage.ts` の `setLocalParticipant` で localStorage 更新
  - DB側の `participants.nickname` も更新（`PATCH /api/participants`）

### C-2. URLからの事前ユーザー登録
- **要望**: ユーザー登録はQRからだけではなく、URLからも事前に登録できるようにする。
- **現状**: 会場QR読み取りを起点に、初回ニックネーム登録して参加者作成。
- **変更後イメージ**: 事前共有URLから先にユーザー登録でき、当日はQR読み取りだけでスタンプ取得できる。
- **検討事項**:
  - 事前登録ページをどのURLにするか
  - プロジェクト別の事前登録URLを発行するか
  - 登録後にどの画面へ遷移させるか
  - 当日QR読み取り時に事前登録済みユーザーとして認識できるか

### C-3. ユーザー情報の紐付け
- **要望**: 現状はニックネーム登録のみなので、将来的にはユーザーを個人情報・外部IDに紐付けたい。
- **現状**: ニックネーム + 端末ローカル + 復元コードが中心。
- **ともや回答（2026-06-21）**:
  - 将来的にはメールアドレス紐付けをしたい。
  - ただし、7月利用時点ではそこまでは必要ないため一旦対応不要。
  - 10月のサーキット時は、公式LINE登録も促す想定なので、LINE紐付けがよさそう。
- **方針メモ**:
  - 7月: 既存のニックネーム + 復元コード運用を継続。
  - 10月: LINE紐付けを第一候補として設計検討。
  - 将来的な汎用機能として、メールアドレス紐付けも候補に残す。
- **検討候補**:
  1. メールアドレス紐付け
  2. LINE紐付け
- **確認したい観点**:
  - ユーザーにとって登録が面倒すぎないか
  - 機種変更・端末変更時の復元がしやすいか
  - なりすまし防止になるか
  - 運営側が取得・管理する個人情報として重くなりすぎないか
  - Supabase/Auth/LINEログイン等の実装コスト
  - 公式LINE登録導線とスタンプラリー登録導線をどうつなげるか

## D. 管理者画面の構成変更

### D-1. プロジェクト詳細ページを4タブに分割
- **現状**: プロジェクト詳細1ページにイベント・特典・取得者・メンバー・削除・再申請が全部集まっていて縦に長い。
- **要望**: 以下の4ページ（またはタブ）に分割
  1. **イベント一覧ページ** — イベントの追加・編集・削除
  2. **特典管理ページ** — 特典段階の設定
  3. **スタンプ取得者・特典取得者管理ページ** — 取得者一覧・CSV書き出し
  4. **メンバー・共同編集者ページ** — メンバー一覧・招待・参加コード表示
- **スコープ**: `app/admin/projects/[id]/page.tsx` の分割。サイドナビまたはタブUIを追加。

### D-2. イベントが増えるとスクロールが多い（ユーザー画面）
- **現状**: スタンプ帳はプロジェクト単位でイベントを縦並びに表示。
- **検討事項**:
  - カード折りたたみ / 横スクロール / ページング
  - プロジェクト絞り込みフィルタ

### D-3. プロジェクトごとの表示画面変更とユーザー登録の紐付け
- **要望**: プロジェクトごとに表示画面を変える。
- **相談事項**: 表示画面を変える場合、ユーザー登録もプロジェクトごとに紐付けるべきか。
- **確認したいこと**:
  - 1人のユーザーが複数プロジェクトに参加する前提にするか
  - ユーザー情報は全プロジェクト共通にするか、プロジェクト単位で別管理にするか
  - プロジェクトごとに必要な登録項目を変えられるようにするか
  - 表示画面だけ変えるのか、ユーザーフローや登録項目まで変えるのか

## E. 認証・セッション

### E-1. 管理者が再ログインを求められる頻度を減らしたい
- **現状の仕組み**:
  - Supabase Auth のアクセストークン有効期限: **1時間**
  - リフレッシュトークン: 最大 **60日**（タブを開き続けていれば自動更新される）
  - `AdminLayout.tsx:16` で `getSession()` を毎ページ呼び出し。セッションが null なら `/admin/login` へリダイレクト。
  - **つまり**: タブを閉じて60日以上放置、またはブラウザのストレージをクリアするとログアウト状態になる。通常の使い方（同じブラウザで継続利用）では再ログインはほぼ不要なはず。
- **要望の確認が必要**: 具体的にどのタイミングで再ログインが発生しているか教えてもらえると原因を特定しやすい（例: スマホでSafariを使っている場合は ITP によって7日でストレージが消える）。
- **対策候補**:
  - iOS Safari の場合は PWA としてホーム画面追加でストレージ保持
  - `supabase.auth.onAuthStateChange` でリフレッシュを確実に行う実装の確認

## 未整理メモ

- QR起点だけだと、当日現地で登録が発生して少し手間になる。
- 事前登録URLがあれば、SNSや公式サイトから先に登録してもらえる。
- 個人情報を持つ場合は、便利になる一方で管理責任も増える。
- 7月時点では、個人情報紐付けは不要。
- 10月サーキットでは、公式LINE登録導線と合わせてLINE紐付けを検討。
- LINE連携はユーザーにとって楽そうだが、実装・審査・運用面を確認したい。
- メール紐付けは将来的な候補として残すが、現場で入力が面倒になる可能性がある。
