## 0. 企画背景 / ともや相談ログ

> ともや = 今回の発注先

### 0-1. きっかけ

- ともやから、10月に大きい企画があり、そこでスタンプラリーをやりたい。できればデジタルで作りたい、という相談があった。
- LINEのスタンプラリー機能もあるが、ともやのやりたいイメージとは違う。おそらく来店ごとのポイント付与くらいで、ワクワク感が足りない。
- 既存のLINEスタンプラリー/来店ポイント機能の代替ではなく、10月企画に合わせた独自体験を作りたい。
- QR読み込みでスタンプ取得できることは前提として、企画性・回遊性・特典導線・デザインの自由度が必要。

### 0-2. プロトタイプ共有と初期反応

- `https://stamp-rally-kappa.vercel.app/` を共有。
- 管理者画面ログイン情報も共有したが、公開メモにはパスワードを記載しない。
- ともやから「すごい」「完璧すぎる」という反応あり。
- まずは仮説ベースで素早くプロトタイプを作成し、触ってもらう流れになった。
- デザインや具体機能は未確定で、フィードバック前提の初期版。

### 0-3. 10月サーキットフェスの想定用途

- 10月にサーキットフェスを実施予定。
- 地域協賛をつけたい。
- ライブハウスと地域の飲食店にスタンプラリーを置き、街全体を回遊してもらえる仕組みにしたい。
- 単なるライブ来場スタンプではなく、ライブハウスと地域飲食店を横断する回遊施策。
- スタンプ地点はライブ会場だけでなく、協賛店舗・飲食店も含まれる想定。
- 地域協賛の価値を出すために、来場者が街を歩き、店舗にも立ち寄る導線が必要。

### 0-4. 開発者側の動機と試用方針

- 42で学んだことをもとに、実際に使ってもらえるプロダクト開発経験を作りたい。
- Emotion時代に作りたいと思っていたものが、今なら実現できると思い、ともやに声をかけた。
- 10月サーキット本番の前に、Emotion関連ライブや街イベントなどで小さく試用する。
- 目的は、机上の仕様決めではなく、実際の現場利用から改善点を見つけること。
- 7月利用範囲と10月本番向け改修を分けて考える。

## 1. 直近利用: 7月の東急関連イベント

### 1-1. 利用予定

- 電話メモ: 東急で7月にあるイベントでスタンプラリーを使わせてほしい。
- 東急確認後、ともやから「7/11・7/12は最低限、QRコードからスタンプを押せて、スタッフが画面タッチすると使用済みになる形だけできれば大丈夫そう」と共有あり。

### 1-2. 7/11・7/12の最低要件

- 来場者がQRコードを読み取ってスタンプを取得できる。
- 特典/利用済み処理は、スタッフが来場者画面を確認し、画面タッチで使用済みにできる。
- 特典なしで、まずは読み取り・取得体験だけ試す運用も可。
- 複雑な個人情報連携やLINE連携は不要。
- デザインは、まず現場で安定して使えるC案ベースが候補。
- スタッフが現場で迷わず使える画面・説明書・運用手順が必要。

### 1-3. 6/21 実験利用ログ

- ともやが実験的に使ってみる予定。ユーザーの声もその場で聞けるとのこと。
- 大型改修後、オーナーごとに管理者申請してからイベント作成できるようになった。
- Emotion内で開催されたイベントで貯まるスタンプと、ストリートライブ系で貯まるスタンプは、ユーザーの中で別枠として貯まるイメージ。
- 責任者がプロジェクト開催申請し、きくがスーパー管理者として承認する運用。
- ともや側でEmotionとストリートライブを別々でスタンプラリー作成できると理解。
- この日は特典なしで、使用感をキャッチアップする目的。
- プロジェクト申請が届き、承認済み。
- 一旦QRコードを読み取ってもらう形だけ使ってみる流れ。
- 課題として「スタンプのQRコード読み取りからしかユーザー登録できないのか？」という疑問が発生。

## 2. 次回以降に変更すること

### 2-1. URLからの事前ユーザー登録

- **要望**: ユーザー登録はQRからだけではなく、URLからも事前に登録できるようにする。
- **現状**: 会場QR読み取りを起点に、初回ニックネーム登録して参加者作成。
- **変更後イメージ**: 事前共有URLから先にユーザー登録でき、当日はQR読み取りだけでスタンプ取得できる。
- **検討事項**:
  - 事前登録ページをどのURLにするか。
  - プロジェクト別の事前登録URLを発行するか。
  - 登録後にどの画面へ遷移させるか。
  - 当日QR読み取り時に事前登録済みユーザーとして認識できるか。

### 2-2. プロジェクトごとに表示画面を変える

- **要望**: プロジェクトごとに表示画面を変える。
- Emotion試用版、ストリートライブ版、東急イベント版、10月サーキット本番版で、色・ロゴ・文言・導線を分ける可能性がある。
- 今後の営業展開では、案件ごとにデザインを切り替えられることが重要。
- プロジェクト単位でテーマカラー、背景、ロゴ、文言、スタンプ画像を変えられる設計が必要。

### 2-3. ユーザー情報の紐付け

- **要望**: 現状はニックネーム登録のみなので、将来的にはユーザーを個人情報・外部IDに紐付けたい。
- **現状**: ニックネーム + 端末ローカル + 復元コードが中心。
- **ともや回答**:
  - 将来的にはメールアドレス紐付けをしたい。
  - ただし、7月利用時点ではそこまでは必要ないため一旦対応不要。
  - 10月のサーキット時は、公式LINE登録も促す想定なので、LINE紐付けがよさそう。

## 3. 公式LINE導線

### 3-1. ともやの整理

- 自社の場合は、公式LINEからスタンプラリーへの動線設計をすれば、顧客をセグメント別に把握した上で宣伝なども合わせてできそう。
- サーキットに向けて、目撃録用の公式LINEを作る予定。

### 3-2. 設計メモ

- LINE連携は、LINEスタンプラリー機能の代替ではなく、公式LINE登録・セグメント把握・宣伝導線として重要。
- 10月サーキットでは、公式LINE登録 → スタンプラリー参加 → セグメント別の告知/宣伝、という導線が検討されている。
- 10月向けには、公式LINEから事前登録URL/プロジェクトURLへ誘導する設計を検討する。
- LINEログインまで実装するか、公式LINE内リンク + ニックネーム/復元コード運用に留めるかを要確認。
- セグメント別把握を行うなら、どの情報を誰が管理するか、個人情報の扱いを整理する必要がある。

## 4. デザイン方針

### 4-1. デザイン案ログ

- 使い方説明書PDF、デザイン検討PDF、デザイン比較PDFを共有。
- ともやの反応:
  - 個人的な好み: B案 or H案。
  - 総合的にはC案がよさそう。
  - オーソドックスでシンプルに行くならC案。
  - イベントとの親和性や雰囲気まで含めるとB案。
- 色はターコイズが有力。
- ターコイズはEmotionカラー/会社カラーっぽく、重要な色。
- サーキット本番版はロゴ・ブランドカラーに合わせた専用テーマが必要。
- サーキットロゴは未公開情報なので、公開リポジトリや外部公開資料に画像そのものを載せない。

### 4-2. 現時点の整理

- C案: オーソドックスでシンプル。初期プロトタイプ/街イベント/東急イベント試用には向いていそう。
- B案: イベントとの親和性や雰囲気まで含めると有力。
- H案: ともや個人の好みとして候補。
- 色: ターコイズが有力。Emotionカラー/会社カラーとの相性がよい。

### 4-3. 次アクション

- 7/11・7/12の東急関連イベント試用はC案ベースで進めるか確認。
- Emotionらしさ・イベント感を重視する版はB案ベースで再検討。
- サーキット本番版はロゴ/ブランドカラーを反映した専用テーマとして別途作る。

## 5. 機能別改修メモ

### A. データ保全

#### A-1. スタンプ取得データを機能改修で消さない
- **現状**: 来場者データは localStorage の `participant_id` + `recovery_code` で識別。DBのデータは消えないが、マイグレーションのカスケード削除で `participant_rewards` が消える既知リスクあり。
- **要望**: 改修のたびにユーザーのスタンプ・特典データが失われないようにしたい。
- **検討事項**:
  - マイグレーション方針の明文化。
  - 削除系DDLは慎重に扱う。
  - `participants` / `event_stamps` / `participant_rewards` に論理削除フラグを検討。
  - 改修前後のデータ確認手順をドキュメント化。

### B. カスタマイズ・見た目

#### B-1. プロジェクトごとのテーマ変更
- `projects` テーブルに `theme_color`、`bg_image_url`、`logo_url` 等のカラム追加を検討。
- 来場者ページでプロジェクトのテーマを動的適用。
- 管理者がプロジェクト詳細から設定できるUIが必要。
- 未公開ロゴや未公開ビジュアルを公開リポジトリに含めない運用。
- 将来的には、運営側がノーコード/低工数で色・ロゴ・文言を変更できる管理画面を検討。

#### B-2. イベントごとの写真/デジタルスタンプ
- `events` テーブルに `stamp_image_url` カラム追加を検討。
- Supabase Storage でのアップロード対応。
- スタンプ帳の各スタンプ枠に画像表示。

#### B-3. 特典チケットタップ時のポップアップ表示
- スタンプ帳で獲得した特典チケットをタップしたとき、スタンプ取得時と同じようなフルスクリーンポップアップを表示したい。
- `RewardTicketModal` が既にあるが、スタンプ帳からの動線・演出を確認する。
- 7/11・7/12向けには、スタッフがタッチして使用済みにする導線を優先確認。

#### B-4. 街回遊・サーキットフェス向けのワクワク感
- 地図/マップ風UI。
- スポット一覧と達成状況。
- ライブハウス・飲食店・協賛店舗などの地点種別表示。
- 「あと◯店舗で特典」などの回遊モチベーション。
- プロジェクトごとのビジュアル/世界観反映。

### C. ユーザー機能

#### C-1. ユーザー名（ニックネーム）の変更
- `/profile` ページはニックネーム表示のみで編集UIが未実装。
- プロフィールページでニックネームをタップ・編集できるようにしたい。
- 保存時は localStorage と DB 側の `participants.nickname` を更新する。

#### C-2. URLからの事前ユーザー登録
- 事前共有URLから先にユーザー登録できるようにする。
- プロジェクト別の事前登録URLを発行するか検討。
- QR読み取り時に事前登録済みユーザーとして認識できるようにする。

#### C-3. ユーザー情報の紐付け
- 7月: 個人情報紐付けは不要。
- 10月: LINE紐付けを第一候補として設計検討。
- 将来的な汎用機能として、メールアドレス紐付けも候補に残す。

### D. 管理者画面 / 運用

#### D-1. プロジェクト詳細ページを4タブに分割
- イベント一覧ページ。
- 特典管理ページ。
- スタンプ取得者・特典取得者管理ページ。
- メンバー・共同編集者ページ。

#### D-2. スタンプ帳のスクロール改善
- カード折りたたみ。
- 横スクロール。
- ページング。
- プロジェクト絞り込みフィルタ。

#### D-3. プロジェクトごとの表示画面変更とユーザー登録の紐付け
- 1人のユーザーが複数プロジェクトに参加する前提にするか。
- ユーザー情報は全プロジェクト共通にするか、プロジェクト単位で別管理にするか。
- プロジェクトごとに必要な登録項目を変えられるようにするか。
- 表示画面だけ変えるのか、ユーザーフローや登録項目まで変えるのか。

#### D-4. ライブハウス・飲食店などスポット管理
- 現在の `events` をそのままスポットとして使うか、`spots` / `locations` のような別概念を追加するか。
- スポット種別: ライブハウス / 飲食店 / 協賛店舗 / その他。
- 各スポットの住所、営業時間、紹介文、画像、外部リンク。
- 店舗ごとのQR発行。
- 店舗ごとの取得数/来訪数CSV。
- 協賛店舗向けに見せられる簡易レポート。

### E. 認証・セッション

#### E-1. 管理者が再ログインを求められる頻度を減らしたい
- Supabase Auth のアクセストークン有効期限: 1時間。
- リフレッシュトークン: 最大60日。
- iOS Safari の場合は PWA としてホーム画面追加でストレージ保持を検討。
- `supabase.auth.onAuthStateChange` でリフレッシュを確実に行う実装の確認。
